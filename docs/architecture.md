# アーキテクチャ

```
Cloud Scheduler(毎朝 JST 8:30 ── SOXAI Ring同期(JST7:00/最大55分・GitHub Actionsのcron遅延あり)の後にバッファを見て起動)
  ↓ HTTP(jobs.run)
Cloud Run Jobs(このリポジトリ、rizap-marketingプロジェクト)
  ├ soxai-ring-runner サービスアカウント(SAキーをSecret Managerからマウント)で
  │   Drive/Sheets/Vertex AI すべて認証
  ├ COND_FOLDER_ID(コンディションチェック親フォルダ)配下の被験者フォルダ
  │   (001_◯◯, 002_◯◯, ...)を走査 → 各「コンディションチェック」スプレッドシートを解決
  ├ 各スプレッドシートから読み取り:
  │   ├ SOXAI_daily      … 日次サマリー(QOLスコア・活動スコア・安静時代謝・活動消費)
  │   └ フォームの回答 1 … 当日アンケート(Q1体調・Q2業務負荷・Q3食事予定・Q5困りごと)
  │      ※SOXAI_detail(5分粒度)は使わない。日次コメントには日次サマリーで十分で、
  │        粒度が細かすぎるデータはノイズ・コスト増になるため
  ├ 前日比・7日平均比・本人平均からのSD逸脱度・欠測日数を計算 → 閾値超えをフラグ化
  ├ AIコメント_ログ(このジョブが管理する永続シート)と突き合わせ、
  │   変化のない日付はキャッシュを再利用、新規/変化ありの日付のみ Vertex AI経由でGeminiを呼び出し
  ├ SOXAI_daily の B列(日付列の隣)にコメント列を書き込み、元データ列は折りたたみ
  └ AIコメント_ログを更新
```

## Geminiに何を読み込ませているか（プロンプトの中身）

日付ごとに、以下だけを渡す。**過去30日の生データは渡さない**。当日のコンディションを
評価するのが目的であり、過去の値に引っ張られたコメントになるのを防ぐため。
過去データは「本人の平常値からどれだけ外れたか」という統計量に圧縮してのみ使う。

| 入力 | 出所 | なぜ渡すか |
|------|------|-----------|
| 4指標の当日値（QOLスコア・活動スコア・安静時代謝kcal・活動消費kcal） | SOXAI_daily | コメントの主対象 |
| 前日比・7日平均比 | SOXAI_dailyから計算 | 直近のトレンド変化を指摘するため |
| 本人平均からのSD逸脱度（±2SDでフラグ） | SOXAI_dailyから計算 | スコア水準は個人差が大きく絶対値では判定できないため、本人基準の異常検知に使う |
| 欠測日数 | SOXAI_dailyから計算 | リング未装着期間を踏まえたコメントにするため |
| 当日アンケート回答（Q1体調・Q2業務負荷・Q3食事予定・Q5困りごと） | フォームの回答 1 | 本人の自己申告。数値との整合/ギャップ（本人は元気なつもりだが数値は低い等）を指摘するため。Q4(リング同期状況)は運用情報なので除外 |

アンケートが未提出の日はセンサーデータのみでコメントを生成し、後から提出された場合は
翌朝の実行で自動的に再生成される（回答内容をキャッシュのハッシュに含めているため）。

## なぜこの構成か

### なぜ「コメントの永続キャッシュ + 再スタンプ」方式か
既存の `rizap-soxai-ring`(GitHub Actions、毎朝JST7:00実行)が `SOXAI_daily` シートを
**毎回delete→再作成**しているため、単純にコメント列を追加するだけでは翌朝の同期で消える。
このジョブは常にSOXAI Ring同期の後に実行し、消えたコメント列を再構築する前提で設計している。
コメントの実体は本ジョブが新設する `AIコメント_ログ` シートに永続化しており、
このシートはSOXAI Ringの削除対象(`SOXAI_daily`/`SOXAI_detail`のみ)ではないため消えない。
過去日付のコメントはキャッシュから再利用するため、Gemini呼び出しは
新規・変化があった日付分のみで済む(コスト・レイテンシを抑制)。

### なぜCloud Run Jobs + Cloud Scheduler(GitHub Actionsではない)か
RIZAP側プロジェクト(rizap-marketing)内で権限・請求・実行がすべて完結するため。
GitHub Actionsで定期実行する場合、REHATCH側GitHubにRIZAPプロジェクトの認証情報を
置くことになり環境の分離が崩れる。日次バッチなのでHTTPエンドポイント不要のJobs形式が最もシンプル。

### なぜVertex AI経由のGeminiか
RIZAP側プロジェクト内の権限(`roles/aiplatform.user`)だけで完結させるため。
APIキーの発行・失効管理が不要になる。Geminiは純正のGoogleモデルなので、
Model Garden経由のパートナーモデルのような追加の利用規約同意も不要。
(コンソール上のブランド名が変わってもAPIは `aiplatform.googleapis.com` で、
モデルは `GEMINI_MODEL` 環境変数でいつでも差し替え可能。既定は gemini-2.5-flash)

### なぜ soxai-ring-runner サービスアカウントを再利用するか
このSAは rizap-soxai-ring が毎朝 `SOXAI_daily` を書き換えるのに使っており、
対象スプレッドシート群へのアクセス実績が既にある。再利用すれば、フォルダ共有の
やり直しもOAuth認可(ブラウザログイン)も不要で、構築が1コマンドで完結する。
本エージェント用に追加するのは Vertex AI呼び出し権限のみ。
