# アーキテクチャ

```
Cloud Scheduler(毎朝 JST 9:00 に起動。SOXAI Ring同期(cron設定JST7:00だがGitHub Actionsの
  遅延で完了実績は8:15〜8:40頃)の後にバッファを見て実行する)
  ↓ HTTP(jobs.run)
Cloud Run Jobs(このリポジトリ、rizap-marketingプロジェクト)
  ├ soxai-runner サービスアカウント(SAキーをSecret Managerからマウント)で
  │   Drive/Sheets/Gemini(Agent Platform) すべて認証
  ├ COND_FOLDER_ID(コンディションチェック親フォルダ)配下の被験者フォルダ
  │   (001_◯◯, 002_◯◯, ...)を走査 → 各「コンディションチェック」スプレッドシートを解決
  ├ 各スプレッドシートから読み取り:
  │   ├ SOXAI_daily      … 日次サマリー(睡眠・ストレス・歩数・活動消費kcal等)
  │   └ フォームの回答 1 … 前日アンケート(Q1体調・Q2業務負荷・Q3食事予定・Q5困りごと)
  │      ※SOXAI_detail(5分粒度)は使わない。日次コメントには日次サマリーで十分で、
  │        粒度が細かすぎるデータはノイズ・コスト増になるため
  ├ 施策開始前平均→開始後平均の変化、当日値の開始後平均からのSD逸脱度、欠測日数を
  │   計算 → 閾値超えをフラグ化
  ├ AIコメント_ログ(このジョブが管理するシート)と突き合わせ、
  │   変化のない日付は再生成せず、新規/変化ありの日付のみ Agent Platform（旧Vertex AI）経由でGeminiを呼び出し
  └ AIコメント_ログに日付ごとのコメントを保存(無ければシート作成、あれば追記・更新)
     ※SOXAI_daily自体には書き込まない(毎朝delete→再作成されるため。
       B列は氏名ID列であり、列挿入はレイアウトを壊すので行わない)
```

## Geminiに何を読み込ませているか（プロンプトの中身）

日付ごとに、以下だけを渡す。**過去30日の生データは渡さない**。当日のコンディションを
評価するのが目的であり、過去の値に引っ張られたコメントになるのを防ぐため。
過去データは「施策開始前平均」「施策開始後平均」「開始後平均からの逸脱度」という
統計量に圧縮してのみ使う（2026-07-11: 前日比・7日平均比・全履歴SD逸脱度から変更）。

比較の軸は2つ:
1. **施策開始前平均 → 施策開始後平均**：施策全体を通じた長期的な変化（施策の効果）
2. **当日の値の、施策開始後平均からのSD逸脱度**：直近の異常検知
   （基準点を「全履歴平均」ではなく「施策開始後の平均」に変更。開始前のデータは
   施策の効果測定にのみ使い、異常検知の基準には混ぜない）

| 入力 | 出所 | なぜ渡すか |
|------|------|-----------|
| 睡眠の実測値（総睡眠時間・深い/浅い/REM睡眠・入眠潜時・睡眠効率・就寝/起床時刻） | SOXAI_daily | 「◯時間しか寝ていない」「就寝がいつもより遅い」「眠りが浅い」等、そのまま声かけに使える指摘の材料 |
| ストレス（日中平均） | SOXAI_daily | 高ストレス日の検知と過ごし方アドバイスの材料 |
| 活動量（歩数・活動消費kcal） | SOXAI_daily | 誰でも意味が分かる活動指標 |
| 施策開始前平均・施策開始後平均 | SOXAI_dailyから計算 | 施策全体の長期的な変化を示すため |
| 開始後平均からのSD逸脱度（±2SDでフラグ） | SOXAI_dailyから計算 | 水準は個人差が大きく絶対値では判定できないため、本人基準の異常検知に使う |
| 欠測日数 | SOXAI_dailyから計算 | リング未装着期間を踏まえたコメントにするため |
| 前日アンケート回答（Q1体調・Q2業務負荷・Q3食事予定・Q5困りごと） | フォームの回答 1 | 本人の自己申告。数値との整合/ギャップ（本人は元気なつもりだが数値は低い等）を指摘するため。Q4(リング同期状況)は運用情報なので除外。常に前日分を使う（後述） |

**渡さないもの**:
- QOLスコア・活動スコア・健康スコア・睡眠スコアなどの**合成スコア** —
  値を見ても「何が良い/悪いのか」を説明できず、具体的なアドバイスに繋がらないため
- 皮膚温・HRV等の生理指標の生値 — 単体では解釈できずノイズになるため
- 過去30日の生データ — 当日の評価が過去に引っ張られるのを防ぐため
  （過去は「施策開始前平均／開始後平均からの逸脱度」という統計量に圧縮してのみ使う）

## 出力フォーマット（OKR＋SBI＋KPT）

`AIコメント_ログ`の「コメント」セルは、以下の構造をテキスト化したものを保存する
（シート自体の列構成は変更していない。セルの中身を構造化しただけ）。

- **O (Objective)**: 施策全体の目標。被験者ごとの数値目標を保持する仕組みが無いため、
  `settings.objective_text`（`OBJECTIVE_TEXT`環境変数で上書き可）の固定文を使う。
  Geminiには生成させない（都度言い換えさせると表記が揺れるため）。
- **KR (Key Results)**: 当日SD逸脱でフラグが立った指標のみを対象にする
  （全11指標を毎日展開すると長大になり実用的でないため）。
  フラグが立った指標が無い日はGeminiを呼ばず、固定の激励文を返す（コスト削減）。
- KRごとに、**SBI**（トレーナー向けの客観的分析: Situation/Behavior/Impact）と
  **KPT**（ゲストへそのまま伝える口語メッセージ: Keep/Problem/Try）を分離して生成する。
  SBIは「事象の分析」、KPTは「伝達するメッセージ」という役割分担とし、
  内容が重複しないようSYSTEM_PROMPTで明示している（`src/comment_generator.py`）。
- Geminiへの入力データそのもの（上表の「Geminiに何を読み込ませているか」）は変更していない。
  出力の構造・分量が変わっただけで、渡す実測値・統計量の範囲は従来通り。
- 出力はGeminiに`response_mime_type="application/json"`でJSONを直接生成させ、
  Python側でパースして整形する。JSON解析に失敗した場合はKRなし（フォールバック文）として扱い、
  ジョブ全体を落とさない（翌日以降、データハッシュ不一致により自動的に再試行される）。
- 出力フォーマット自体を変更した際は、`main.py`の`COMMENT_FORMAT_VERSION`を上げて
  ハッシュに含める。数値・アンケートが変わっていない日でも、フォーマット変更時は
  全件が再生成対象になる。

補足:
- 睡眠が計測されていない日（総睡眠min=0）は0ではなく欠測として扱い、本人の平常値を歪めない
- 就寝/起床時刻は時刻数値に変換して比較し（深夜0時台は24h+として扱う）、
  「普段より◯時間遅い」を判定できるようにしている
- アンケートは常に**前日分**を参照する（実行時刻(7時台〜)には当日分が全員提出済みとは
  限らないため、日によって当日/前日が混在しないよう統一。前日分は実行時点で確定して
  いるため、後から提出されて再生成が必要になることもない）
- 施策開始日は、被験者ごとの環境変数ではなく**マスター名簿（`00_被験者名簿`、シート`被験者名簿`）**
  から取得する（`ROSTER_SHEET_ID`。既定値は確定済みの名簿ファイルID）。名簿の`soxai_id`列が
  各被験者のSOXAI_dailyの`ユーザーID`列と完全一致するため、名簿を1回読み込むだけで
  全被験者分を引ける。各被験者スプレッドシートへの転記は不要。名簿に行が無い/開始日が
  未入力の被験者は「施策開始後の変化」を提示せず、当日のSD逸脱度のみ（全履歴基準）で評価する

## なぜこの構成か

### なぜ出力先が `AIコメント_ログ` シート（SOXAI_dailyには書き込まない）か
既存の `rizap-soxai-ring`(GitHub Actions、毎朝JST7:00実行)が `SOXAI_daily` シートを
**毎回delete→再作成**しているため、SOXAI_dailyに列を足しても翌朝の同期で消える。
また SOXAI_daily のB列は氏名ID列であり、列を挿入するとレイアウトが変わってしまう。
そこで本ジョブが新設する `AIコメント_ログ` シートを唯一の出力先とする。
このシートはSOXAI Ringの削除対象(`SOXAI_daily`/`SOXAI_detail`のみ)ではないため消えず、
無ければ作成・あれば追記なので既存データに影響を与えない。
生成済みの日付は再生成しないため、Gemini呼び出しは新規・変化があった日付分のみで済む。

### なぜJST9:00の1回実行か
SOXAI Ring同期(GitHub Actions)はcron設定こそJST7:00だが、GitHub Actionsのスケジュール
遅延で実際の開始は7時台後半、完了は8:15〜8:40頃(直近10回の実測)。9:00なら同期完了後に
バッファを持って1回で実行できるため、シンプルなcron1本にしている。
ジョブは「日付ごとに、生成済みで変化が無ければスキップ・新規/変化ありのみ生成」という
冪等な作りのため、まれに同期が9:00に間に合わなかった日はその日の生成は行われず、
翌朝の実行で前日分もまとめて生成される(欠落はしない)。同期の遅延が常態化した場合は
`SCHEDULE`環境変数(またはgcloud schedulerの更新)で時刻を後ろへずらすだけでよい。

### なぜCloud Run Jobs + Cloud Scheduler(GitHub Actionsではない)か
RIZAP側プロジェクト(rizap-marketing)内で権限・請求・実行がすべて完結するため。
GitHub Actionsで定期実行する場合、REHATCH側GitHubにRIZAPプロジェクトの認証情報を
置くことになり環境の分離が崩れる。日次バッチなのでHTTPエンドポイント不要のJobs形式が最もシンプル。

### なぜAgent Platform（旧Vertex AI）経由のGeminiか
RIZAP側プロジェクト内の権限(`roles/aiplatform.user`)だけで完結させるため。
APIキーの発行・失効管理が不要になる。Geminiは純正のGoogleモデルなので、
Model Garden経由のパートナーモデルのような追加の利用規約同意も不要。
※Vertex AIは2026年5月に「Gemini Enterprise Agent Platform」（コンソール表記: Agent Platform）に
改称されたが、APIエンドポイント(`aiplatform.googleapis.com`)・IAMロール・SDKは変更なし。
モデルは `GEMINI_MODEL` 環境変数でいつでも差し替え可能（既定は gemini-2.5-flash）。

### なぜ soxai-runner サービスアカウントを再利用するか
このSAは rizap-soxai-ring が毎朝 `SOXAI_daily` を書き換えるのに使っており、
対象スプレッドシート群へのアクセス実績が既にある。再利用すれば、フォルダ共有の
やり直しもOAuth認可(ブラウザログイン)も不要で、構築が1コマンドで完結する。
本エージェント用に追加するのは Gemini(Agent Platform)呼び出し権限のみ。
